# Extension Development

How to build a firnline extension package — the developer's side of the plugin
system. For installing and managing extensions in a deployed instance, see
[../guides/installing-extensions.md](../guides/installing-extensions.md).

## What an extension is

An **extension** is one pip-installable Python package that contributes
vertical slices to firnline without touching kernel code. It registers via
Python entry points and may bundle a **schema module** (class/enum definitions
for TerminusDB) alongside one or more **plugin implementations** (ingest
sources, extractors, query tools, capture handlers, trigger evaluators, action
executors, or indexer plugins).

## Package layout

```
firnline-ext-example/
├── pyproject.toml
└── src/
    └── firnline_ext_example/
        ├── __init__.py          # expose SCHEMA_MODULE_DIR if providing schema
        ├── manifest.json        # schema module manifest
        ├── schema.json          # class/enum definitions (JSON array)
        ├── models.py            # generated by firnline-schema codegen (never hand-edit)
        ├── extract.py           # ExtractorPlugin
        ├── tools.py             # ToolPlugin
        ├── sources.py           # IngestSourcePlugin
        ├── capture.py           # CaptureHandler
        ├── evaluators.py        # TriggerEvaluator
        ├── executor.py          # ActionExecutor
        └── indexer.py           # IndexerPlugin
```

All source lives under `src/<package_name>/`. The build backend is
`hatchling`:

```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[tool.hatch.build.targets.wheel]
packages = ["src/firnline_ext_example"]
```

Use `[tool.uv.sources]` with `{ workspace = true }` to depend on
`firnline-core`.

## Entry points and protocols

Extensions register plugins via Python entry-point groups in `pyproject.toml`.
Each group maps to a specific protocol in `firnline_core.plugins`. For
complete signatures and protocol details, see
[../reference/entry-points.md](../reference/entry-points.md).

### Available groups

| Entry-point group | Protocol | Used by |
|---|---|---|
| `firnline.schema_modules` | Directory with `manifest.json` + `schema.json` | bootstrap, firnline-schema |
| `firnline.ingestd.sources` | `IngestSourcePlugin` | ingestd |
| `firnline.ingestd.extractors` | `ExtractorPlugin` | ingestd |
| `firnline.queryd.tools` | `ToolPlugin` | queryd |
| `firnline.captured.handlers` | `CaptureHandler` | captured |
| `firnline.triggerd.evaluators` | `TriggerEvaluator` | triggerd |
| `firnline.effectd.executors` | `ActionExecutor` | effectd |
| `firnline.notifyd.channels` | `NotificationChannel` (deprecated) | effectd (auto-adapted) |
| `firnline.indexed.indexers` | `IndexerPlugin` | indexed |

Example registration in `pyproject.toml`:

```toml
[project.entry-points."firnline.schema_modules"]
time_management = "firnline_ext_time_management:SCHEMA_MODULE_DIR"

[project.entry-points."firnline.ingestd.extractors"]
time_management_extractor = "firnline_ext_time_management.extract:plugin"

[project.entry-points."firnline.queryd.tools"]
time_management_tools = "firnline_ext_time_management.tools:plugin"
```

### Collision rules

Duplicate registrations within a group (same capture kind, same tool name,
same trigger type, same executor kind, same indexed class) across active
plugins are **fatal startup errors**. Extension authors must ensure global
uniqueness.

## Schema module authoring

A schema module consists of two files:

- **`manifest.json`** — metadata: `name`, `version` (semver), `depends_on`
  (module dependencies with version ranges), `models_target` (dotted Python
  path for codegen output), `exports` (class/enum `@id` values), `description`.
- **`schema.json`** — a JSON array of TerminusDB class/enum definitions in
  WOQL schema format. The `@context` object is owned by the `core` module
  only; extensions must not include it.

Every class/enum listed in `exports` must have an `@documentation` key with
a non-empty `@comment` — enforced by `firnline-schema compose` (L3 lint).
Also required:

- **`@metadata.label_field`** on every exported concrete `Entity` subclass
  (L4 validation).
- **`@metadata.anchor_field`** on every concrete class implementing `Anchored`
  (L5 validation).

Full format details: [../reference/schema-modules.md](../reference/schema-modules.md).

## Codegen — typed models for extensions

Third-party extensions can generate typed Pydantic models from their schema
module definitions:

1. Declare `models_target` in `manifest.json`, e.g.
   `"models_target": "firnline_ext_myapp.models"`.
2. When `firnline-schema compose` discovers the extension via the
   `firnline.schema_modules` entry point, it records the module-to-target
   mapping.
3. Run `firnline-schema codegen` — it locates the package via `importlib`,
   classifies classes by owning module, and emits `models.py` into each
   package's tree.
4. Import the generated models from your extension's `models.py`.

Kernel modules land in `firnline_core.generated/`. Extension models land in
the extension's own package. No central registry, no dependency on
`firnline-core` for the generated code.

## Worked example: firnline-ext-people

`firnline-ext-people` is a first-party extension in `extensions/` that
demonstrates the pattern:

```
extensions/firnline-ext-people/
├── pyproject.toml        # entry points: schema_modules + extractors + indexers
└── src/firnline_ext_people/
    ├── __init__.py        # SCHEMA_MODULE_DIR = str(importlib.resources.files(…))
    ├── manifest.json      # name: "people", depends_on: [places], models_target
    ├── schema.json        # Person, Contact (@subdocument)
    ├── models.py          # generated (never hand-edit)
    └── extract.py         # PeopleLinkingPlugin — provides known person names during linking
```

Key patterns from the extension code:

- `__init__.py` exposes the schema module directory path so the entry point
  can reference it as a string attribute:
  ```python
  import importlib.resources
  SCHEMA_MODULE_DIR = str(importlib.resources.files("firnline_ext_people"))
  ```
- The extractor plugin (`extract.py`) implements the `ExtractorPlugin`
  protocol with `name`, `requires`, `requires_classes`, and `produces`.
- `requires_classes` declares TDB class `@id` values the plugin needs
  from other modules (checked against the `SchemaModule` registry at startup).

## Migration notes

If you previously relied on core constructs that have been removed:

- **`Remindable`** is gone — use `Triggerable` (from the `triggers` module)
  or define your own markers.
- **`anchor_at`** no longer exists — use `@metadata.anchor_field` on the
  concrete class implementing `Anchored` instead.
- **`Provenance.source`** is gone — use `Entity.derived_from: Set<Source>`.
- **`firnline.notifyd.channels`** is deprecated — migrate to
  `firnline.effectd.executors` (the legacy group is auto-adapted with kind
  `notify:<name>`, but will be removed after one release cycle).

## Service startup and plugin lifecycle

All services boot through the shared `PluginHost` in `firnline-core`:

1. **Discover** plugins for the service's entry-point group.
2. **Validate** structural conformance against the protocol.
3. **Fetch** the `SchemaModule` registry from TerminusDB.
4. **check_requirements** — module semver ranges + `requires_classes`
   against installed module exports.
5. **Collision check** — duplicate keys are fatal.
6. **Select** active plugins; failures are skipped or fatal per `HostPolicy`.

Each service has its own `HostPolicy` (broken_entry_point_fatal,
zero_active_fatal, strict, tdb_unavailable_fatal). See
[../concepts/plugin-system.md](../concepts/plugin-system.md) and
[../concepts/architecture.md](../concepts/architecture.md) for the per-service
values.

## Testing extensions

Extension tests live under `<extension>/tests/` and follow the same
pytest + pytest-asyncio conventions as the rest of the repo. Common patterns:

- **Discovery tests** — verify `manifest.json` and `schema.json` format,
  entry-point resolution, plugin instantiation.
- **Protocol validation** — instantiate the plugin object and assert
  `name`, `requires`, `produces`, `kinds`, etc.
- **Logic tests** — exercise extractor/build/execute logic with test fixtures,
  mock TDB clients, or `FunctionModel` for deterministic LLM responses.

See [testing.md](testing.md) for general test conventions.
